From an expert technical architect perspective, the flows have been revised to precisely match the full requirement set. Each major flow is now strictly isolated in its own card with vertical step-by-step sequencing for clear containment and readability. Pagination is explicitly shown in Flow 4 (ClassService + ClassRepository).

1. Auth & Registration Flow (Red)

Client submits credentials via Postman / App
→
POST /api/auth/register
RegisterRequest {Username, Password, Role}
→
AuthAPIController → AuthService.RegisterAsync()
→
AuthResponse {Token, Expiry, Username, Role}
Client submits login credentials
→
POST /api/auth/login
LoginRequest {Username, Password}
→
AuthAPIController → AuthService.LoginAsync()
→
AuthResponse {Token, Expiry, Username, Role} (or 401)

2. Class & Booking Operations (Blue)

Instructor creates/updates/deletes class (role-checked via JWT)
→
POST /api/classes
CreateClassRequest {Title, Description, InstructorId, ScheduledAt, Capacity}
→
ClassesAPIController → ClassService.CreateClassAsync() → AccountAPIClient.InstructorExistsAsync()
→
ClassResponse (or 400/403)
Member books / cancels class (role-checked via JWT)
→
POST /api/bookings
BookingRequest {ClassId}
→
BookingsAPIController → BookingService.BookClassAsync() (capacity + duplicate check)
→
BookingResponse (or 409)
Member views own bookings / booked classes (paginated)
→
GET /api/bookings
GET /api/classes
→
BookingsAPIController / ClassesAPIController → BookingService.GetBookingsForMemberAsync() / ClassService.GetClassesForMemberAsync()
→
PagedResult<ClassResponse> / List<BookingResponse>

3. Cross-Service Validation (Green)

ClassService validates instructor before class creation/update
→
GET /api/users/exists/{id}
→
AccountAPIClient → UsersAPIController → AuthService.UserExistsAsync()
→
bool (true/false)
ClassService fetches InstructorName for denormalization
→
GET /api/users/{id}
→
AccountAPIClient → UsersAPIController → AuthService.GetUserByIdAsync()
→
User (InstructorName extracted)

4. Dynamic Search (Yellow)

Authenticated client requests filtered/paginated class list
→
GET /api/classes/search?date=…&instructorName=…&page=…&pageSize=…
→
ClassesAPIController → ClassService.SearchClassesAsync() (pagination) → ClassRepository.GetPagedAsync()
→
PagedResult<ClassResponse>

Domain Models ▼
ModelFieldsPurpose Note

DTOs ▼
DTOFieldsPurpose Note
LoginRequestUsername, PasswordInbound DTO for login endpoint. Never stored.
RegisterRequestUsername, Password, RoleInbound DTO for registration. Password is hashed immediately; plain text is never persisted.
AuthResponseToken (string), Expiry, Username, RoleReturned to caller on successful register or login. Contains the signed JWT + expiry for client-side token management.
Service Classes ▼
IAuthService / AuthService
Central orchestrator for all identity operations
▾
Purpose: Orchestrates registration, login, and user queries. Depends on IUserRepository and IJwtTokenService. Keeps controllers thin.
Function SignaturePurpose
Task<AuthResponse> RegisterAsync(RegisterRequest req)Validates username uniqueness, hashes password via IPasswordHasher, persists User, mints JWT and returns AuthResponse. Returns 409 Conflict if username taken.
Task<AuthResponse?> LoginAsync(LoginRequest req)Looks up user by username, verifies password hash (constant-time), delegates to IJwtTokenService to mint a signed JWT. Returns null if credentials invalid → controller maps to 401.
Task<User?> GetUserByIdAsync(Guid id)Retrieves User entity by PK. Used by UsersAPIController to serve instructor-existence checks from ClassAPI.
Task<IEnumerable<User>> GetUsersByRoleAsync(Role role)Returns all users of a given role. Powers the GET /api/users/instructors endpoint.
Task<bool> UserExistsAsync(Guid id)Lightweight Any() probe — avoids loading full entity. Called by ClassAPI before assigning an instructor to a new class.
IJwtTokenService / JwtTokenService
Encapsulates all JWT creation and validation logic
▾
Purpose: Keeps cryptographic JWT concerns out of AuthService. Lives in Infrastructure/Security/ — single place to change signing algorithm, secret, or expiry policy.
Function SignaturePurpose
string GenerateToken(User user)Builds a signed JWT with claims: sub (UserId), name (Username), role (Role). Uses HS256 with secret from appsettings.json. Returns compact serialized token string.
ClaimsPrincipal? ValidateToken(string token)Parses and validates a token for internal inspection (e.g., token refresh). Returns null if invalid or expired.
DateTime GetTokenExpiry()Returns configured token lifetime end. Surfaced in AuthResponse so clients know when to re-authenticate without trial-and-error.
IUserRepository / UserRepository
EF Core data abstraction for User entities
▾
Purpose: Abstracts EF Core operations. Enables unit testing by allowing mock injection without hitting the database.
Function SignaturePurpose
Task<User?> GetByIdAsync(Guid id)Fetch single user by PK.
Task<User?> GetByUsernameAsync(string name)Used during login and registration uniqueness check.
Task<IEnumerable<User>> GetByRoleAsync(Role role)Filters users by role for instructor list endpoint.
Task<bool> ExistsAsync(Guid id)Existence probe — avoids loading full entity for lightweight checks.
Task AddAsync(User user)Persists a new User entity; calls SaveChangesAsync.
IPasswordHasher / PasswordHasher
Thin wrapper around BCrypt — keeps hashing algorithm swappable
▾
Purpose: Isolates hashing algorithm behind an interface. Lives in Infrastructure/Security/ — swap BCrypt for Argon2 without touching AuthService.
Function SignaturePurpose
string Hash(string plainText)Returns BCrypt hash of the plain-text password. Called at registration time.
bool Verify(string plainText, string hash)Constant-time comparison to prevent timing-based side-channel attacks. Called during login.
Controllers & Endpoints ▼
AuthAPIController  [Route: api/auth]
Public identity endpoints — no [Authorize] required
▾
Purpose: Entry point for unauthenticated users. Accepts credentials, delegates to AuthService, returns JWT tokens that unlock all other services.
MethodRouteAuthService CallPurpose
POST /api/auth/register Public AuthService.RegisterAsync() Accepts RegisterRequest DTO. Returns 201 + AuthResponse (JWT) on success. Returns 409 Conflict if username already taken.
POST /api/auth/login Public AuthService.LoginAsync() Accepts LoginRequest DTO. Returns 200 + AuthResponse on success. Returns 401 Unauthorized if credentials are invalid.
UsersAPIController  [Route: api/users]
Internal endpoints consumed by ClassAPI over HTTP
▾
Purpose: Serves instructor data to ClassAPI's AccountAPIClient. All endpoints require a valid JWT.
MethodRouteAuthService CallPurpose
GET /api/users/{id} [Authorize] AuthService.GetUserByIdAsync() Returns User details (excluding PasswordHash). Used by ClassAPI to fetch InstructorName.
GET /api/users/exists/{id} [Authorize] AuthService.UserExistsAsync() Lightweight probe. ClassAPI calls this before assigning an instructor to a new class.
GET /api/users/instructors [Authorize] AuthService.GetUsersByRoleAsync(Instructor) Returns list of all instructors. Used to populate class-creation dropdowns.

05 — Cross-Cutting Concerns
Infrastructure & Shared Components
▼

Components applied across both microservices — middleware, auth policies, HTTP clients, and tooling. Shared project contains DTOs, exceptions and middleware used by both AccountAPI and ClassAPI.

GlobalExceptionHandlerMiddleware
Both Services (Shared)

Catches unhandled exceptions, returns ProblemDetails JSON. Maps domain exceptions to HTTP codes.

RequestLoggingMiddleware
Both Services (Shared)

Logs HTTP method, path, status code, elapsed time. Configurable per environment.

JwtBearerMiddleware
ClassAPI (+ AccountAPI) (Shared)

Validates Bearer token. Extracts UserId and Role claims for controller injection via HttpContext.User.

Authorization Policies
ClassAPI

"InstructorOnly" and "MemberOnly" policies applied via [Authorize(Policy="...")] on controllers.

HttpClientFactory (typed)
ClassAPI

Registers AccountAPIClient. Avoids socket exhaustion vs raw HttpClient instantiation.

Swagger / SwaggerUI
Development

Configured with JWT Bearer authentication scheme for easy testing of secured endpoints.

Docker / docker-compose
Deployment

Multi-stage Dockerfiles. Orchestrates both services, sets consistent environment variables.

FitnessBooking.Shared
Shared Project

Contains DTOs (AuthResponse, ClassResponse, PagedResult), Exceptions (DomainException), Middleware (GlobalExceptionHandlerMiddleware, RequestLoggingMiddleware, JwtBearerMiddleware).

Key Architectural Decisions ▼
InstructorName Denormalization

InstructorName stored on FitnessClass avoids cross-service HTTP calls on every search query. Written once at class creation.

MemberId from JWT

MemberId extracted from HttpContext.User claims, never from request body. Prevents booking on behalf of others.

Repository Pattern

Enables full unit test coverage without DB setup. Interfaces in Domain/Interfaces/; implementations in Infrastructure/Persistence/Repositories/. Controllers depend on services; services depend on repos.

BookedCount Atomicity

BookedCount increment and Booking insert happen in single EF SaveChanges transaction to prevent race conditions.

Clean Architecture Layering

Each service is split into Domain → Application → Infrastructure → Presentation. Security concerns (JWT, password hashing) are isolated in Infrastructure/Security/. Dependency direction is always inward.
